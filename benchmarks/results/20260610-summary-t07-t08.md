# Benchmark results — 2026-06-10, t07/t08 (medium tasks), n=3 per cell

> **Validity note (added 2026-06-10):** these runs were measured with `verify.cjs` visible in the work dir (pre-hidden-oracle runner). Visible tests inflate agent pass rates 20-60% (arXiv:2602.10975), so cells here likely overstate success in BOTH modes and are not comparable to post-fix results. See benchmarks/README.md "Hidden oracle".

These are the first cells sized to where Maestro's Decision Gate could
plausibly bite: t07 is a multi-file feature across a 9-file app
(new module + CLI wiring + config + docs), t08 a cross-cutting
error-handling refactor across 10 modules plus a barrel. n=3 remains a
small sample — medians, no significance claims; rows comparable within
this table only.

- CLI: Claude Code (claude.exe, native build), model `sonnet`
- Isolation: per-run `CLAUDE_CONFIG_DIR` (no global config, hooks, or MCP in either cell)
- ON = repo `AGENTS.md` + `CLAUDE.md` copied into work dir; OFF = absent
- Per-run cap: `--max-budget-usd 2.0` (never hit)
- Fixture sanity (pre-run): pristine copy fails `verify.cjs`, hand-fixed gold copy passes, both tasks
- Raw data: `20260610-043338-claude-sonnet.json`

## Reading (n=3, still small) — did Maestro earn its overhead?

Plainly: **not on these cells, at this model tier.** Sonnet OFF already
solves both medium tasks 3/3, so there is no failure for the doctrine
to prevent — ON pays +$0.017-0.032 median cost and +5-19% median wall
for the same 100% success. One directional note in Maestro's favor:
on t08 the ON cell used fewer median turns (24 vs 26) — consistent with
the edit-safety rules steering a more systematic sweep — but it did not
translate into lower cost or time.

Honest implication: at sonnet-on-deterministic-fixtures scale, success
parity is the ceiling and overhead is the measurable effect. Cells
where pass-rate deltas could appear need either harder tasks (failure
rate > 0 OFF) or weaker models; that escalation is future work, not a
claim.
